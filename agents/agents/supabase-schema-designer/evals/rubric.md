# supabase-schema-designer ルーブリック

## マルチテナント設計（0.0-1.0）
- すべての業務テーブルに tenant_id UUID NOT NULL が定義されている
- RLS ポリシーが SELECT / INSERT / UPDATE / DELETE の全操作に定義されている
- RLS ポリシーが auth.jwt() を使用してテナントを識別している
- tenant_id に対するインデックスが作成されている

## スキーマ品質（0.0-1.0）
- 主キーが gen_random_uuid() を使用した UUID である
- created_at / updated_at が全テーブルに付与されている
- 外部キーに適切な ON DELETE アクション（CASCADE / SET NULL / RESTRICT）が設定されている
- ENUM 型が CREATE TYPE で定義されている
- 正規化が適切に行われ、不要なデータ重複がない

## インデックス戦略（0.0-1.0）
- 外部キーカラムにインデックスが作成されている
- 検索頻度の高いカラムに複合インデックスが定義されている
- JSONB カラムに GIN インデックスが使用されている（該当する場合）
- インデックスに合理的な根拠が説明されている

## トリガー・関数（0.0-1.0）
- updated_at 自動更新トリガーが全テーブルに設定されている
- 業務要件に応じた追加トリガー（監査ログ等）が実装されている
- トリガー関数が SECURITY DEFINER / INVOKER を適切に使用している

## マイグレーションSQL（0.0-1.0）
- Up / Down マイグレーションが両方生成されている
- Down マイグレーションが Up の逆順で完全にロールバックできる
- 依存関係の順序（ENUM → テーブル → インデックス → トリガー）が正しい
- Supabase Migrations のファイル命名規則（タイムスタンプ_説明）に従っている

## Mermaid ER図（0.0-1.0）
- erDiagram 記法が正しく、Mermaid でレンダリング可能である
- 主キー（PK）・外部キー（FK）が明示されている
- テーブル間のリレーション（||--o{等）が正確に表現されている
- すべてのテーブルが図に含まれている

## 曖昧さの処理（0.0-1.0）
- 不明な要件が AMBIGUITY: / ASSUMPTION: 形式で明示されている
- 設計上の前提が設計サマリーに記載されている
- スコープ外の提案が「追加推奨事項」に分離されている
